# ASCP Source Archive

> **Transparency, Accountability, and Preservation**

This organization contains an archive of the ASCP source tree.

ASCP is built upon open-source projects including Android Open Source Project (AOSP), LineageOS, Pixel Os, and numerous other community contributions. Credit for all original work belongs to the respective authors, maintainers, and contributors.

No ownership over any upstream work is claimed.

---

# Why This Archive Exists

The Android ecosystem was built on open-source principles.

Android became successful because developers could inspect, improve, audit, modify, learn from, and preserve source code. Open source allows trust to be backed by verification rather than assumptions.

Open source provides:

- Independent security auditing
- Community review
- Verification of developer claims
- Faster bug discovery and resolution
- Long-term project preservation
- Greater accountability

Trust is important.

**Verification is better.**

Open source enables verification.

---

# Why Source Transparency Matters

Android ROMs operate with full control over a user's device.

A ROM can:

- Access user data
- Modify partitions
- Control update mechanisms
- Execute privileged operations
- Change security behavior
- Affect device stability

Because of this level of access, users should never be expected to blindly trust any developer, team, or project.

Transparency allows users and developers to independently verify what software is actually doing.

The concern is not private development branches.

Many projects develop features privately before publishing them.

The concern arises when users are expected to run software while being unable to independently inspect the code responsible for its behavior.

Projects such as Infinity X maintain publicly available source repositories while still managing their own development workflow. Their source remains available for auditing, verification, preservation, and future development.

This is a healthy approach.

The issue is not private development.

The issue is permanently private source code for software that users are expected to trust with complete control over their devices.

# Lessons From Project Elixir

The Android community has already witnessed why transparency matters.

One of the most notable examples was the Project Elixir controversy.

Following community investigations, concerns were raised regarding code capable of wiping user data under specific conditions. The incident generated significant discussion throughout the Android development community and eventually resulted in moderation actions on XDA Forums.

Relevant references:

Relevant references:

- [ElixirOS to break and wipe your device](https://xdaforums.com/t/elixiros-to-break-and-wipe-your-device.4672456/)
- [[CLOSED] Why were Project Elixir ROM threads banned?](https://xdaforums.com/t/closed-why-were-project-elixir-rom-threads-banned.4681395/)

The purpose of referencing Project Elixir is not to attack any project or developer.

The lesson is broader:

> Software running with complete control over a user's device should be publicly reviewable.

Transparency protects:

- Users
- Developers
- Maintainers
- Projects
- The community as a whole

Users should not be forced to rely exclusively on trust.

They should be able to verify.
